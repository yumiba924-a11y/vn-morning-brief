# Tier2 発火判定 設計書 — 掲示板バズ → 監視引き上げ

> ステータス：**設計のみ（未実装）**。実装は下記「前提条件」が揃うまで着手しない。
> センサー（social collector）は稼働中で、`social_history/buzz_daily.csv` に毎営業日データが貯まる。

## 目的

掲示板バズ（social collector）を、三層ユニバースの **Tier2 発火判定** に接続する。
VN31–100 の70銘柄を平時は無視し、**バズが平常比で跳ねた銘柄だけ**監視に引き上げる。

## 接続の全体像

3つの部品でできている。

| 部品 | 中身 | 状態 |
|---|---|---|
| **地図** | VN30/VN100 実測（VN30=**73.5%**、増分**14pp**・1銘柄平均**0.2pp**）→ 監視範囲を3層に区切る根拠 | 静的・確定済み |
| **センサー** | `social_history/buzz_daily.csv` → 銘柄別バズを毎営業日記録 | 稼働中 |
| **配線（本設計書）** | センサーの値が跳ねたら Tier2 に引き上げる判定ロジック | 本書で設計 |

## スパイク判定の勘所（実装時の芯）

1. **平常比で見る（絶対数で見ない）。**
   FPT は普段から142件、VJC は1件。絶対数で閾値を引くと大型株ばかり発火する。
   各銘柄の**自分の移動平均**（例：直近10営業日）との比で判定する。

2. **clean ベースで見る（broadcast 煽り除外後の件数）。**
   VIC のように投稿72でも半分が煽りのケースがあるので、`is_broadcast=false` の件数
   （＝`volume_n_clean`）で平常比を取る。

3. **閾値は data-driven で後決め。**
   数日〜2週の分布を見てから、平常比×何倍を発火とするか決める（例：直近平均の2.5倍超）。
   **今は閾値を仮置きせず、分布観察を待つ。**

4. **量と質を分けて使う。**
   - **量スパイク**（type0 が跳ねる）＝リテール過熱の早期シグナル → **発火のトリガー**
   - **質**（type1 のニュース/専門家投稿）＝発火した銘柄の「なぜ動いたか」の解釈材料 → **発火後に引く**

## 発火の対象範囲

| 層 | 扱い |
|---|---|
| **VN30** | 常時監視。バズは補助解釈に使う（**発火判定には使わない**） |
| **VN31–100** | バズスパイクを**主要な発火トリガー**にする（**ここが本命**） |
| **VN101以下** | 対象外。バズが跳ねても文脈参照に留める |

## 出力の接続先

発火した銘柄リスト → 既存 `build_brief.py` の「1コールで JSON 要約」型に流す。
発火銘柄の**質テキスト**＋**代表的な量投稿**を渡して、日本語で「**なぜ発火したか**」を要約させる。
→ 週次レポートの **Tier2 セクションの生素材**になる。
